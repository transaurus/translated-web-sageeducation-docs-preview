### 💡 Dockerを採用する理由

Dockerがすべてのユーザーの好みではないことは理解していますが、このアプローチは当プロジェクトの設計と運用効率において中心的な役割を果たしています。私たちはDockerへのコミットメントをプロジェクトの基本方針と捉えており、異なるデプロイ方法を求める方にはコミュニティ主導の代替案を探ることを推奨します。

#### **Q: ロゴとブランディングをカスタマイズする方法は？**

**A:** テーマ、ロゴ、ブランディングのカスタマイズは、**[Enterprise License](https://docs.openwebui.com/enterprise)** で提供されるエンタープライズ機能をご利用いただけます。

エンタープライズソリューションとブランディングカスタマイズの詳細については、営業チームまでお問い合わせください: 📧 **sales@openwebui.com**

#### **Q: なぜサインアップが必要なのですか？ データはどこに送信されますか？**

**A:** セキュリティ強化のため、管理者ユーザーとしてのサインアップが必要です。これにより、Sage WebUIが外部アクセスに晒された場合でも、データの安全性が保たれます。重要な点として、全てのデータはローカルに保持されます。当社がデータを収集することはありません。サインアップ時に提供される情報は全てサーバー内に留まり、デバイス外に送信されることはありません。ユーザーのプライバシーとセキュリティを最優先し、常にデータの管理権限をユーザーが保持できるようにしています。

#### **Q: Dockerコンテナからホストのサービスに`localhost`で接続できないのはなぜですか？**

**A:** Dockerコンテナ内で`localhost`と指定すると、ホストマシンではなくコンテナ自身を指します。このネットワーク上の違いが重要です。コンテナからホストで動作しているサービスに接続するには、`localhost`の代わりにDNS名`host.docker.internal`を使用してください。この特別なDNS名はDockerによって認識され、コンテナ内からホストへの接続を可能にします。これにより、通常の`localhost`のスコープ制限を回避できます。

#### **Q: ホストのサービスをDockerコンテナからアクセス可能にするにはどうすればよいですか？**

**A:** ホストで動作しているサービスをDockerコンテナからアクセス可能にするには、これらのサービスを`127.0.0.1`（localhost限定）ではなく、IPアドレス`0.0.0.0`を使用して全てのネットワークインターフェースでリッスンするように設定してください。この設定により、Dockerコンテナを含む任意のIPアドレスからの接続が許可されます。ただし、外部アクセスが可能な環境ではセキュリティ上の影響に注意が必要です。ファイアウォールや認証などの適切なセキュリティ対策を実施することでリスクを軽減できます。

#### **Q: Sage WebUIが更新されません。コンテナを再プル/再起動しても変化がありません**

**A:** Sage WebUIを更新するには、新しいDockerイメージをプルするだけでは不十分です。更新が反映されない理由と確実に更新する方法を説明します：

1. **Dockerイメージの更新**: `docker pull ghcr.io/Startr/AI-WEB-openwebui:main`コマンドはDockerイメージを更新しますが、実行中のコンテナやそのデータは更新しません。
2. **Dockerボリュームの永続データ**: Dockerボリュームはコンテナのライフサイクルとは独立してデータ（チャット履歴など）を保持するため、更新後もデータが維持されます。
3. **更新の適用**: 既存のコンテナを削除（ボリュームは削除せず）し、更新されたイメージと既存のボリュームをアタッチして新しいコンテナを作成することで、更新を確実に適用できます。

この手順により、アプリケーションを更新しながらデータを安全に保持できます。

#### **Q: コンテナを削除する必要があるのですか？データが失われませんか？**

**A:** よくある懸念ですが、Dockerボリュームを正しく使用していれば、コンテナを削除してもデータは失われません。その理由は：

- **ボリュームによるデータ保護**: Dockerボリュームはコンテナのライフサイクル外でデータを永続化するように設計されています。データがボリュームに保存されている限り、コンテナに何が起きても影響を受けません。
- **安全な更新プロセス**: Sage WebUIを更新する際、古いコンテナを削除し、更新されたイメージで新しいコンテナを作成しても、ボリュームに保存されたデータには影響しません。重要なのは`docker volume rm`のようなコマンドで明示的にボリュームを削除しないことです。

正しい更新手順（新しいイメージのプル、ボリュームを削除せずに古いコンテナの削除、更新されたイメージと既存のボリュームを使用した新しいコンテナの作成）に従うことで、アプリケーションコードは更新されつつ、データは変更されず安全に保持されます。

#### **Q: ディストリビューション提供のDockerと公式Dockerパッケージ、どちらを使用すべきですか？**

**A:** Sage WebUIの実行には、ディストリビューション提供版ではなく公式Dockerパッケージの使用を推奨します。公式パッケージは最新の機能、バグ修正、セキュリティパッチが頻繁に更新され、最適なパフォーマンスと安全性が保証されます。さらに、`host.docker.internal`のような重要な機能（ディストリビューション版では利用できない場合がある）をサポートしています。この機能はDockerコンテナ内での適切なネットワーク設定と接続性に不可欠です。

公式Dockerパッケージを選択することで、さまざまな環境間で一貫した動作が保証され、トラブルシューティングのサポートがより信頼性の高いものとなり、最新のDockerの進化にもアクセスできます。また、公式パッケージには広範なDockerコミュニティとリソースが揃っており、遭遇する可能性のある問題に対する豊富な情報とサポートが提供されます。

Sage WebUIを実行するために必要なすべてのもの（データを含む）は、ユーザーの管理下とサーバー環境内に留まります。これは、プライバシーとセキュリティに対する私たちの取り組みを強調するものです。公式Dockerパッケージのインストール手順については、Docker公式ドキュメントサイトの[Docker Engineのインストール](https://docs.docker.com/engine/install/)ガイドを参照してください。

#### **Q: DockerでGPUサポートは利用可能ですか？**

**A:** DockerでのGPUサポートは利用可能ですが、プラットフォームによって異なります。公式には、Docker for WindowsとLinux上のDocker EngineでGPUサポートが提供されています。Docker Desktop for LinuxやMacOSなどの他のプラットフォームでは、現在のところGPUサポートは提供されていません。この制限は、GPUアクセラレーションを必要とするアプリケーションにとって重要です。最適な体験とGPU機能を活用するためには、公式にGPU統合をサポートしているプラットフォームでDockerを使用することをお勧めします。

#### **Q: Sage WebUIはなぜDockerの使用を強調しているのですか？**

**A:** Dockerを使用するという決定は、一貫性を確保し、依存関係を分離し、さまざまな環境での展開を簡素化する能力に由来しています。Dockerは互換性の問題を最小限に抑え、基盤となるシステムに関係なくWebUIを迅速に起動および実行するプロセスを合理化します。これは、プロジェクトのメンテナーによる戦略的な選択であり、Dockerには学習曲線があることを認めつつも、展開とメンテナンスにおける利点が大きいことを認識しています。Dockerがすべての人の好みではないかもしれないことを理解していますが、このアプローチは私たちのプロジェクトの設計と運用効率の中核をなすものです。私たちは、プロジェクトのDockerへのコミットメントを基本的な側面と見なし、異なる展開方法を探求したい人々にはコミュニティ主導の代替案を探ることを奨励しています。

#### **Q: 私の展開では音声認識（STT）と音声合成（TTS）が機能しません。なぜですか？**

**A:** 展開における音声認識（STT）と音声合成（TTS）サービスの機能性は、正しく動作するためにHTTPSを必要とする場合があります。現代のブラウザはセキュリティ対策を強化しており、STTやTTSなどの特定の機能を安全なHTTPS接続下でのみ動作するように制限しています。展開がHTTPSを使用するように設定されていない場合、これらのサービスは期待通りに機能しない可能性があります。展開がHTTPS経由でアクセス可能であることを確認することで、これらの問題を解決し、STT/TTS機能の完全な機能性を有効にすることができます。

#### **Q: Sage WebUIにはなぜ組み込みのHTTPSサポートがないのですか？**

**A:** すべてを包含するソリューションとしてHTTPSサポートを含めることの要望は理解していますが、そのようなアプローチではユーザーベースの多様なニーズに十分に対応できないと考えています。プロジェクト内に直接HTTPSを実装することは柔軟性を制限し、すべてのユーザーの特定の要件や好みに合致しない可能性があります。すべての人が自分のユニークな環境に合わせてセットアップを調整できるようにするため、本番展開におけるHTTPS終端の実装はユーザーに委ねています。この決定により、より大きな適応性とカスタマイズが可能になります。HTTPSの設定に関する公式ドキュメントは提供していませんが、コミュニティメンバーが要望に応じてガイダンスを提供し、彼らの経験に基づいた洞察と提案を共有する場合があります。

#### **Q: 更新/再起動/新しいソフトウェアをインストールした後、Sage WebUIが動作しなくなりました！**

**A:** 更新後や新しいソフトウェアのインストール後にSage WebUIが起動しない場合、特にバックエンドの依存関係に仮想環境を使用しなかった直接インストールアプローチに関連している可能性があります。直接インストールは、既存の依存関係を変更する更新や新しいインストールなど、システム環境の変更に対して敏感です。競合を避け、安定性を確保するために、バックエンドの`requirements.txt`依存関係を管理するために仮想環境を使用することをお勧めします。これにより、Sage WebUIの依存関係が他のシステムパッケージから分離され、このような問題のリスクが最小限に抑えられます。

#### **Q: 更新/再起動後、ログインができなくなり、新しいアカウントを作成する必要がありました。そして、すべてのチャットが消えています。**

**A:** この問題は通常、Dockerコンテナが`/app/backend/data`用のボリュームをマウントせずに作成された場合、または指定されたSage WebUIボリューム（通常は例で`sage-open-webui`と命名）が誤って削除された場合に発生します。Dockerボリュームは、コンテナのライフサイクルを超えてデータを永続化するために重要です。再起動後に新しいアカウントを作成する必要がある場合、既存のデータが存在するボリュームをアタッチせずに新しいコンテナを起動した可能性が高いです。データ損失を防ぐためには、Docker runコマンドに正しいデータロケーションを指すボリュームマウントを含めるようにしてください。

#### **Q: ログインを試みましたができず、新しいアカウントを作成したところ、管理者によるアカウントの有効化が必要だと言われました。**

**A:** この状況は、最初のセットアップ時に作成された初期管理者アカウントのパスワードを忘れた場合に発生します。最初のアカウントは自動的に管理者アカウントとして指定されます。管理者アカウントにアクセスできない状態で新しいアカウントを作成すると、管理者による有効化が必要になります。Sage WebUIへのシームレスなアクセスと管理のためには、初期管理者アカウントの資格情報を失わないことが重要です。管理者アカウントを回復する手順については、[管理者パスワードのリセット](troubleshooting/password-reset)ガイドを参照してください。

#### **Q: Sage WebUIがSSLエラーで起動できないのはなぜですか？**

**A:** Sage WebUIの起動時に発生するSSLエラーは、SSL証明書が存在しないか、[huggingface.co](https://huggingface.co/)の設定が誤っていることが原因である可能性があります。この問題を解決するには、[hf-mirror.com](https://hf-mirror.com/)などのHuggingFaceのミラーを設定し、Dockerコンテナ起動時にエンドポイントとして指定することができます。Docker runコマンドで`-e HF_ENDPOINT=https://hf-mirror.com/`パラメータを使用してHuggingFaceミラーのアドレスを定義してください。例えば、Docker runコマンドを以下のように変更できます。

```bash
docker run -d -p 3000:8080 -e HF_ENDPOINT=https://hf-mirror.com/ --add-host=host.docker.internal:host-gateway -v sage-open-webui:/app/backend/data --name sage-open-webui --restart always ghcr.io/Startr/AI-WEB-openwebui:main
```
